# EMU-Stu 实验室开源与协作指南

本指南由 **EMU-Stu 组织管理委员会** 制定，旨在规范学校各实验室之间的代码协作、项目管理及技术沉淀。所有加入 EMU-Stu 组织的成员需严格遵守本流程。

## 一、 项目周期

项目从构思到成熟分为五个阶段：**筹划、立项、开发（内测）、沉淀（公测）、归档（毕业）**。

---

## 二、 筹划与立项

### 1. 课题分类

- **公开课题**：旨在全校或全网开源的项目，鼓励多实验室联合开发。

- **私有课题**：涉及实验室核心技术或未发表论文的研究，需严格控制可见性。

### 2. 立项流程

1. **初始创建**：项目负责人需先在**个人 GitHub 账号**下创建仓库并进行基础开发。

2. **申请迁移**：当项目完成度达到计划的 **50%** 时，在 [EMU-Stu/DOPMC](https://www.google.com/search?q=https://github.com/EMU-Stu)（建议你建一个专门的管理仓）提交 Issue 进行立项申请。

3. **仓库迁移**：
   
   - **公开项目**：审核通过后，负责人将仓库迁移（Transfer）至 EMU-Stu 组织。
   
   - **私有项目**：由负责人联系 **Organization Owner（组织创建者）** 手动创建私有仓库，并由 Owner 完成 Team 权限分配。

4. **权限分配**：迁移完成后，项目负责人将被授予该仓库的 **Maintain** 权限。

---

## 三、 协作与代码规范（核心：PR 模式）

为防止代码冗余和“乱改”现象，EMU-Stu 强制执行 **Pull Request (PR)** 协作模式。

### 1. 协作权限逻辑

- **公开项目**：全员拥有 **Read** 权限。非核心贡献者必须通过 **Fork** 模式参与。

- **私有项目**：仅限该实验室 Team 成员可见。

### 2. 提交规范（所有成员强制执行）

任何成员（包括项目负责人）不得直接推送（Push）代码至 `main` 分支。

1. **Fork 仓库**：将组织仓库 Fork 到个人账号下。

2. **本地开发**：在个人 Fork 仓库中修改代码。

3. **发起 PR**：通过 GitHub 页面向组织原仓库发起 Pull Request。

4. **代码审核（Review）**：
   
   - 各实验室项目 PR 由该 **实验室 Team 负责人（Maintainer）** 进行审核。
   
   - 审核内容包括：代码逻辑、注释规范、是否存在敏感数据泄露。

5. **合并**：审核通过后，由负责人点击 `Merge` 合并代码。

---

## 四、 实验室隔离策略

为了保护各实验室的学术隐私，组织执行以下硬性规则：

- **基础权限屏蔽**：组织的 `Base permissions` 设置为 **None**。

- **定向授权**：私有仓库仅对所属实验室 Team 开放。

- **禁止私建私有仓**：普通成员无权在组织内自建私有仓库，必须经由管理委员会审核后统一创建，以确保权限审计。

---

## 五、 项目阶段标注

为了清晰标识项目状态，README 标题后需注明：

- **【实验性】**：早期开发阶段，代码可能频繁重构。

- **【稳定版】**：已通过内测，可供其他实验室参考学习。

- **【归档】**：项目已结项或论文已发表，进入只读维护状态。

---

## 六、 行为准则

1. **尊重知识产权**：引用他人代码需注明出处。

2. **严禁泄密**：不得将包含服务器密钥、内网 IP、私密数据集的代码推送到公开仓库。

3. **友好交流**：Issue 和 PR 评论区严禁人身攻击。

---

> **致项目负责人**：如果你准备好了迁移你的第一个项目，请立即联系 Owner 并按上述流程发起 PR 协作测试。
